# Бэклог реализации дашборда

## Сводка

Строим десктоп-дашборд (Electron + Python/FastAPI), который парсит документы из
`docs/` как источник структуры (модули курса, трекер практики, 90-дневный
backlog, RAG MVP), показывает «где я сейчас и над чем работать» и прогресс по
трём измерениям: Теория → Практика → Проекты. Прогресс хранится отдельно
(SQLite), исходные `.md` не модифицируются. Начинаем с вертикального среза на
сущности «модули курса»: парсинг → хранилище → API → экран, затем добираем
остальные представления и главный экран-агрегатор.

## Решения по архитектуре

- **Мост Electron ↔ Python:** локальный FastAPI-сервис. Electron (main process)
  поднимает Python как дочерний процесс при старте, фронт ходит по REST на
  `127.0.0.1`. Причина: чистое разделение слоёв, бэкенд тестируется автономно,
  парсинг/агрегация остаются на Python.
- **UI-стек:** React + TypeScript + Vite, сборка десктопа через `electron-vite`.
  Стилизация — Tailwind CSS с кастомным токен-набором (без дефолтной палитры).
  Причина: зрелый стек под информационно-плотный дашборд, быстрый HMR, типобезопасность.
- **Парсинг Markdown:** на Python — `python-frontmatter` для YAML-шапки +
  `markdown-it-py`/`mistune` для таблиц, чек-листов и заголовков. Obsidian-разметка
  (wikilinks, callouts, mermaid) парсится терпимо: callout `> [!success]` → текст
  критерия, mermaid/wikilinks игнорируются для отображения.
- **Хранилище прогресса:** SQLite (через `sqlalchemy` или `sqlite3`) в
  пользовательской папке Electron (`app.getPath('userData')`). Структура из
  Markdown накладывается на состояние прогресса по стабильным ID. Исходные
  документы — read-only.
- **Идентификация сущностей:** стабильный ID = `<источник>:<якорь>` (например
  `course:module-5`, `backlog:week-3:item-2`), вычисляется детерминированно из
  позиции/заголовка при парсинге, чтобы прогресс не «отвязывался» при перечитывании.

## Модель данных

Сущности (структура парсится из файла, состояние — из SQLite):

- **CourseModule** ← `course-ml-training.md`
  - поля: `id`, `number`, `title`, `niche`, `deadline_hint`, `study_points[]`,
    `task`, `acceptance_criterion` (из callout `[!success]`).
  - состояние: `status` (`not_started|in_progress|done|optional`), `note`.
- **PracticeRow** ← `ml-training-practice-log.md` (таблица прогресса)
  - поля: `id`, `module_ref`, `artifact`, `metric_criterion`, `link`.
  - состояние: `status`, `artifact_url`, `journal_entries[]` (по шаблону записи).
- **BacklogItem** ← `solo-90-day-execution-backlog.md`
  - поля: `id`, `phase` (1–4 / 5–8 / 9–12), `week` (1–12), `text`.
  - состояние: `done` (bool), `done_at`.
- **NinetyDayMetric** ← тот же файл, раздел «Метрики 90 дней»
  - поля: `id`, `text`, `target`. состояние: `current_value`, `done`.
- **MvpCriterion** ← `rag-mvp-spec.md` («Критерии приёмки MVP» + «Backlog разработки» 0.1–0.4)
  - поля: `id`, `group` (acceptance / v0.1..v0.4), `text`. состояние: `done`.
- **DodItem** ← `aimarket-dashboard.md` («Definition of Done»)
  - поля: `id`, `text`. состояние: `done`.

Вычисляемое (на бэкенде, не хранится):

- **DimensionProgress** — три измерения:
  - Теория = доля модулей со `status in (in_progress→частично, done)` по `study_points`.
  - Практика = доля `PracticeRow` с `status=done` (засчёт по правилам трекера).
  - Проекты = доля выполненных `MvpCriterion (group=acceptance)` + `DodItem`.
- **CurrentFocus** — первый невыполненный элемент в порядке: текущая неделя
  backlog → текущий незакрытый модуль курса → ближайший незакрытый критерий MVP.

## Отложено / под вопрос

- Двусторонняя запись прогресса обратно в Markdown — решено хранить отдельно (SQLite),
  чтобы не модифицировать исходные документы.
- Авторизация, мультипользователь, облачная синхронизация — нет в описании проекта.
- Рендер mermaid-диаграмм и wikilinks как интерактивных элементов — не требуется для MVP,
  парсятся/игнорируются как обычный текст.
- Реальная интеграция RAG-пайплайна в дашборд — это отдельный проект (`rag-mvp-spec`),
  дашборд лишь трекает его критерии.
- Авто-обновление при внешнем изменении `.md` (file watcher) — под вопрос, пока хватает
  перечитывания при старте/по кнопке.
