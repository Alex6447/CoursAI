# Промт для генерации бэклога реализации

> Назначение: дать AI-агенту (Claude Code) этот промт целиком, чтобы он
> исследовал контекст проекта и выдал детальный, структурированный бэклог
> задач для реализации Electron-дашборда управления ML-обучением.
>
> Как использовать: открой проект в Claude Code, включи **plan mode**,
> вставь блок ниже (от `<task>` до конца) как первое сообщение. Effort: `high`.

---

<role>
Ты — ведущий инженер-планировщик. Твоя задача — превратить описание проекта
в исполняемый бэклог задач, по которому другой инженер (или ты сам в режиме
implementation) сможет собрать продукт без доуточнений. Ты не пишешь код на
этом шаге — ты проектируешь план.
</role>

<context>
Проект — десктоп-дашборд для управления личным процессом ML-обучения и
создания портфельных кейсов. Дашборд наглядно показывает «где я сейчас и над
чем работать», и прогресс по нескольким измерениям: Теория → Практика →
Создание проектов.

Источники истины (прочитай их ПОЛНОСТЬЮ перед планированием):
- @docs/project.md — базовое описание и стек (Electron, бэкенд на Python, современный UI).
- @docs/course-ml-training.md — 10 модулей курса: статусы, критерии зачёта, сроки, матрица требований.
- @docs/ml-training-practice-log.md — трекер практики: таблица прогресса, шаблон записи, правила зачёта.
- @docs/solo-90-day-execution-backlog.md — недельный план на 90 дней с чек-листами и метриками.
- @docs/rag-mvp-spec.md — спецификация капстоуна (RAG MVP): scope, критерии приёмки, eval-набор.
- @docs/aimarket-dashboard.md — карта проекта, Definition of Done, еженедельный цикл.

Это данные, которые дашборд должен агрегировать и визуализировать. Документы
сейчас в Markdown с Obsidian-разметкой (frontmatter, wikilinks, callouts,
mermaid, таблицы) — учитывай это как формат входных данных.
</context>

<technical_constraints>
- Оболочка: Electron (desktop). Бэкенд-логика: Python. Связь Electron ↔ Python
  выбери и обоснуй (например, локальный HTTP/IPC-сервис на FastAPI, либо
  python-shell/child_process) — это должно быть отдельной ранней задачей-решением.
- UI-стек на твоё усмотрение, но обоснуй выбор одним абзацем (фреймворк,
  сборщик, стилизация). Цель — современный, читаемый дашборд, а НЕ редакторский лендинг.
- Источник данных на старте — Markdown-файлы из docs/. Предусмотри парсинг
  frontmatter, таблиц и чек-листов. Не привязывайся к внешней БД, пока это не
  обосновано задачей.
- Платформа разработки — Windows 11. Учитывай это в командах и путях.
</technical_constraints>

<frontend_aesthetics>
Для проработки дизайна используй скилл `frontend-design`.
NEVER use generic AI-generated aesthetics: avoid overused fonts (Inter, Roboto,
Arial, system fonts), cliched purple-gradient schemes, and cookie-cutter layouts.
Это инструмент/дашборд — выбери чёткую сетку, спокойную информационную палитру,
хорошую типографику для данных и аккуратные микро-взаимодействия. Перед версткой
предложи 2–3 различных визуальных направления (фон / акцент / шрифт + одна строка
обоснования) как отдельную задачу, чтобы пользователь выбрал одно.
</frontend_aesthetics>

<instructions>
Действуй строго по шагам:

1. ИССЛЕДУЙ. Прочитай все шесть документов из <context>. Выпиши сущности данных,
   которые дашборд визуализирует (модули курса и их статусы, артефакты практики,
   недельные задачи backlog, критерии приёмки MVP, метрики DoD). Не предполагай —
   опирайся на то, что реально есть в файлах.

2. ЗАДАЙ ВОПРОСЫ. Если есть развилки, которые меняют архитектуру (например:
   нужно ли редактировать данные из дашборда или только просматривать; нужна ли
   запись прогресса обратно в Markdown), используй инструмент AskUserQuestion и
   задай НЕ более 4 ключевых вопросов. Не задавай очевидное.

3. СПРОЕКТИРУЙ МОДЕЛЬ ДАННЫХ. Опиши, как сущности из документов отображаются в
   структуры, которые рендерит UI (что парсится из Markdown, что вычисляется,
   например агрегированный процент прогресса по трём измерениям).

4. СГЕНЕРИРУЙ БЭКЛОГ. Разбей реализацию на эпики, а эпики — на атомарные задачи.
   Упорядочи их по зависимостям так, чтобы каждая следующая задача опиралась на
   готовые предыдущие. Первый эпик — каркас (scaffold) и решение по Electron↔Python.
   Двигайся от «вертикального среза» (одна сущность от парсинга до экрана) к полноте.

5. ПРОВЕРЬ. Перед выдачей убедись: каждая задача имеет проверяемый критерий
   готовности; нет «висящих» зависимостей; покрыты все сущности данных из шага 1;
   нет задач сверх описанного scope.
</instructions>

<scope_discipline>
- Планируй ровно то, что нужно для работающего дашборда по описанию. Не добавляй
  фич сверх: ни авторизации, ни облачной синхронизации, ни мультипользователя,
  ни аналитики/телеметрии — если это явно не следует из документов.
- Не проектируй «на будущее». Три похожие задачи лучше преждевременной абстракции.
- Если функциональность под вопросом — вынеси её в раздел «Отложено / под вопрос»,
  а не включай в основной бэклог.
</scope_discipline>

<output_format>
Выведи результат как Markdown-файл со следующей структурой:

## Сводка
2–4 предложения: что строим и каким срезом начинаем.

## Решения по архитектуре
Маркированный список ключевых развилок и выбранного варианта с краткой причиной
(Electron↔Python мост, UI-стек, парсинг Markdown, хранение прогресса).

## Модель данных
Сущности и их поля; что парсится из какого файла; что вычисляется.

## Бэклог
Для каждого эпика — заголовок и его задачи. Каждая задача строго в таком виде:

- **[E<эпик>-T<номер>] Название задачи** — `размер: S|M|L`
  - Что сделать: одно-два предложения.
  - Файлы/области: какие файлы создаются или меняются (предположительные пути).
  - Зависит от: список ID задач (или «—»).
  - Критерий готовности: конкретная проверка (что увидеть/запустить, чтобы
    считать задачу выполненной).

Порядок задач = порядок выполнения.

## Отложено / под вопрос
То, что осознанно не входит в первую реализацию, с одной строкой причины.

Не добавляй разделов сверх перечисленных. Не пиши код реализации — только план.
</output_format>

<style>
Пиши по-русски, технические термины и идентификаторы — в оригинале. Будь
конкретным и буквальным: формулируй задачи так, чтобы инженер с минимальным
контекстом мог их выполнить без доуточнений. Без воды и общих рассуждений.
</style>
